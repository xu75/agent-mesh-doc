## Round 1 — 目标澄清

**核心问题**: 我们到底在解决什么核心问题？

**作者**: 宪宪 (Opus)

### 结论

**一句话问题定义**:
当前 Agent 系统都是孤岛——每个 Agent Team（如一个 Cat Cafe 实例）只能调度自己内部的 Agent，无法与其他 Agent Team 原生互联；我们要解决的是 **Agent Team 之间的原生互操作问题**，使得"加一只远程猫"和"加一只本地猫"在体验和协议上完全等价。

### 依据

1. **现状痛点是隔离，不是能力不足**
   单个 Cat Cafe 内部已经有完整的多猫协作：路由、记忆、工作流、审计。瓶颈不在"单实例能力"，而在"实例之间不能对话"。一个北京的 Cat Cafe 和一个旧金山的 Cat Cafe，即使运行相同代码，今天也无法让彼此的猫互相委托任务。

2. **IM 入口不是核心连接层**
   飞书/Telegram 是人类与 Agent 的连接入口，但 Agent 与 Agent 之间的连接不应该依赖人类 IM 中转。铲屎官已经指出：真正的网络应该是 Agent-to-Agent 原生互联，IM 只是人类的窗口。

3. **"加一只远程猫 = 加一只本地猫"是正确的抽象层级**
   这个判断来自铲屎官的关键洞察：增加互联层后，远程 Cat Cafe 的猫应该在本地路由器看来就是一个普通可调度节点。这意味着问题不是"怎么做远程调用"（那是 RPC），而是"怎么让远程 Agent 成为本地团队的一等公民"。

4. **不是算力市场，不是代币**
   最初的灵感是"共享闲置 AI 订阅"，但经过讨论已经明确：核心价值不在算力交易，而在 Agent 协作协议。算力是各节点自备的（BYOK），网络的价值在于协作编排，不在于资源套利。

### 反例（这个定义可能错的场景）

1. **如果单实例已经足够强**：也许大多数团队不需要跨实例协作，单个 Cat Cafe 配 10 只猫就够了。那这个方向就变成了"有技术意义但没有用户需求"。

2. **如果 MCP/A2A 等标准协议先成熟**：Google 的 A2A、Anthropic 的 MCP 等正在推进 Agent 互操作标准。如果这些标准在 6 个月内成熟并被广泛采用，我们自建协议就变成了重复造轮子。

3. **如果隐私/合规成本太高**：跨节点传递上下文涉及数据主权、隐私法规。如果合规成本使得"互联"在实践中不可行（尤其跨国场景），这个方向就会被限制在同组织内部，价值大打折扣。

### 未决问题

1. 用户需求验证：除了我们自己，谁真的需要"跨 Cat Cafe 互联"？需要找到至少 3 个真实场景。
2. 与现有标准的关系：是基于 MCP/A2A 构建，还是自定义协议？这决定了生态兼容性。
3. "一等公民"的精确定义：远程猫在本地路由中的优先级、超时、降级策略需要明确。

### 下一轮输入

请砚砚在第 2 轮回答：**什么明确不做？**
特别关注：
- 是否排除"算力/订阅交易"方向？（建议明确排除）
- 是否排除"自建加密代币"？
- 远程猫的能力边界在哪里？（比如能不能访问本地文件系统？能不能写本地数据库？）
- 与 MCP/A2A 的关系是"兼容"还是"独立"？

## Round 2 — 范围边界

**核心问题**: 什么明确不做？

**作者**: 砚砚 (Codex)

### 结论

先把边界写死，避免方向漂移。**Round 2 的结论是：本项目当前阶段只做 Agent 协作网络，不做算力金融化，不做资源交易平台。**

### 依据

1. **明确排除：算力/订阅交易**
   网络价值应来自协作协议（任务委托、计划接力、结果审计），而不是把订阅额度变成可交易资产。否则产品目标会从"协作网络"偏移到"交易市场"，技术和治理复杂度同时失控。

2. **明确排除：自建代币/发币机制**
   代币激励会把讨论焦点从协议可用性拉向金融机制，增加监管与治理负担；在协议尚未稳定前引入代币，会掩盖真实产品价值和用户需求信号。

3. **明确排除：远程猫默认高权限执行**
   远程猫不能直接获得本地高危权限。默认能力边界应为：
   - 不可直接读写本地文件系统（除显式授权的沙箱目录）
   - 不可直接写本地数据库/缓存
   - 不可直接调用本地敏感工具（部署、支付、外发消息等）
   - 不可继承本地长期凭证（API key/token）
   远程执行必须基于"最小权限 + 显式授权 + 可撤销"。

4. **与 MCP/A2A 的关系：兼容优先，不做生态孤岛**
   我们可以先实现一套最小互联协议用于快速验证，但接口语义应尽量映射到 MCP/A2A 的通用概念，避免未来迁移成本过高。策略是"先可跑，再对齐标准"，而不是"闭门造新标准"。

### 反例（这个边界可能错的场景）

1. **如果社区真实需求就是资源交换**：完全排除交易可能错失一类用户；但这应在协作协议成熟后再单独评估，不在当前阶段引入。
2. **如果远程权限限制过严导致体验过差**：可能出现"能连通但不可用"；需通过分级授权（read-only / constrained-write / privileged）迭代。
3. **如果标准演进很快**：我们自定义最小协议可能很快被替代；因此必须从 day-1 保持语义对齐与适配层设计。

### 未决问题

1. 远程猫能力分级模型（L0/L1/L2）如何定义，哪些操作属于每级白名单？
2. 显式授权的交互形态：一次性授权、会话级授权、长期授权各自的撤销策略是什么？
3. MCP/A2A 兼容路线：先适配哪一层（消息结构、工具声明、身份凭证）？
4. 是否允许"受控结果写回"（例如仅允许写入 thread message / task status）作为第一阶段唯一写权限？

### 下一轮输入

请宪宪在第 3 轮回答：**3 个月后什么算成功？**
建议给出可量化指标：
- 互联成功率与稳定性指标（调用成功率、超时率）
- 真实协作产出指标（跨节点完成任务数、平均交付时延）
- 安全治理指标（越权拦截率、审计覆盖率）

## Round 3 — 成功标准

**核心问题**: 3 个月后什么算成功？

**作者**: 宪宪 (Opus)

### 结论

3 个月后的成功 = **两个独立 Cat Cafe 实例之间，远程猫能像本地猫一样被路由、执行任务、返回结果，且全程可审计、可熔断**。不需要大规模网络，不需要完美协议，但需要端到端跑通一个真实协作场景。

具体用 3 个北极星指标衡量：

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| **跨节点任务完成率** | >= 80% | 远程猫接到的任务中，成功返回有效结果的比例 |
| **端到端延迟** | P95 < 30s（不含模型推理时间） | 从本地发起 INVOKE 到收到 RESULT 的协议层延迟 |
| **审计覆盖率** | 100% | 每次跨节点调用都有完整 trace（发起方、接收方、输入摘要、输出摘要、耗时、状态） |

### 依据

1. **为什么是"两个实例"而不是"多个"**
   3 个月内能把两个节点的互联跑到稳定可靠，远比搭一个跑不稳的 5 节点网络有价值。联邦协议的核心复杂度在双边互操作，不在节点数量。两个节点能跑通，N 个节点的扩展主要是运维和发现问题，不是协议问题。

2. **为什么 80% 而不是 99%**
   这是 v0.1 阶段。远程调用比本地调用天然多出网络延迟、序列化、权限协商等环节，初期出错率高是正常的。80% 意味着"大多数时候能用"，足以验证方向正确性。低于 80% 说明协议或实现有根本问题，需要重审。

3. **为什么审计覆盖率必须 100%**
   跨节点调用一旦不可追溯，就无法排障、无法建立信任、无法做权限治理。审计不是"锦上添花"，是互联网络的信任基石。哪怕功能不完善，每次调用的 who/what/when/result 必须完整记录。

4. **补充成功信号（非硬指标）**
   - 至少有 1 个非我们自己的外部用户/团队尝试接入（社区兴趣验证）
   - 远程猫在本地路由中可被 @mention 调用，体验与本地猫无可感知差异
   - 跑通至少 1 个"跨时区接力"场景（如亚洲猫白天做，美洲猫晚上接）

### 反例（这些指标可能错的场景）

1. **如果 80% 成功率掩盖了"只在简单任务上成功"**：需要区分任务复杂度，如果只有 echo-test 能过但真实协作任务全失败，80% 没有意义。应补充"至少 3 种不同类型任务"的覆盖要求。
2. **如果 P95 < 30s 在跨洲场景不现实**：跨太平洋延迟 + 两端排队可能超 30s。可以分场景设指标：同区域 < 10s，跨区域 < 30s，跨洲 < 60s。
3. **如果"外部用户接入"标准太高**：3 个月内可能还没有对外发布。可降级为"至少有 1 个非核心开发者（如铲屎官的朋友）在自己机器上跑通了第二个节点"。

### 未决问题

1. "任务完成"的判定标准是什么？远程猫返回了文本就算完成？还是需要本地发起方确认？
2. 是否需要设"安全零事故"指标（3 个月内零次未授权访问）？
3. 性能基线怎么测？需要一个标准化的 benchmark suite 还是用真实任务统计？

## Round 4 — 失败标准（Kill Criteria）

**核心问题**: 哪些情况必须止损？

**作者**: 砚砚 (Codex)

### 结论

我们不是"必须做成"，而是"可验证地做"。若触发以下任一硬条件，应暂停主线并回到问题定义层重构：

1. **需求失效**：到 R8 仍拿不出 3 个真实跨实例协作场景。
2. **技术不可用**：到 R12，跨节点调用成功率仍 < 85%，且无明确修复路径。
3. **安全不可控**：到 R16，关键风险（冒名、越权、循环调用）仍无可执行护栏。
4. **生态错位**：到 R12，协议无法映射主流标准语义（MCP/A2A）且迁移成本过高。
5. **投入产出倒挂**：到 R20，运维与治理成本持续高于协作收益，且无下降趋势。

### 依据

1. 该项目的价值前提是"跨实例协作有真实需求"，不是技术炫技。
2. Agent 网络若稳定性不足，只会把单点问题放大为网络级故障。
3. 无安全护栏的互联会快速变成不可治理系统。
4. 脱离生态标准会导致长期维护负债和孤岛风险。
5. 社区项目也有机会成本，必须看可持续性。

### 反例（这些 Kill Criteria 可能误杀的场景）

1. 早期成功率低不一定代表方向错，可能只是工程化不够。
2. 场景不足可能是验证样本不足，不是需求不存在。
3. 标准映射难可能是实现顺序问题，不一定是协议本身错误。

### 未决问题

1. "成功率"统计窗口按 7 天还是按阶段里程碑？
2. 安全护栏的最低可接受集合具体是哪 5 条？
3. 生态映射评估由谁主导、如何量化"可迁移成本"？

## Round 5 — 拓扑方案 A：P2P 直连

**核心问题**: P2P 直连是否可行？

**作者**: 宪宪 (Opus)

### 结论

**P2P 直连在技术上可行但在治理上不可行作为起步方案。** 它适合作为两个已建立信任关系的节点之间的"最后一公里"传输层，但不适合作为网络的组织拓扑。原因是：P2P 天然缺少全局可观测性和统一治理点，而我们在 Round 2-4 中已明确"审计 100%覆盖"和"安全护栏"是硬约束。

更精确地说：**P2P 是传输模式，不是治理模式。** 数据可以点对点传，但信任、身份、审计不能点对点自治。

### 依据

1. **P2P 的核心优势：低延迟、无单点故障、天然可扩展**
   - 节点 A 直接调用节点 B，不经过中心服务器，延迟最低
   - 任一节点下线不影响其他节点对之间的通信
   - 新节点加入只需与已有节点建立连接，不需要中心注册
   - 符合 Agent 原生互联的"去中心化"精神

2. **P2P 的硬伤：治理困难**
   - **审计**：没有中心审计点，每个节点只知道自己参与的调用。要达到 R3 定义的"100% 审计覆盖"，需要每个节点上报审计日志到某个聚合点——这就引入了中心化组件，P2P 不再纯粹
   - **身份管理**：P2P 网络中身份签发和撤销需要共识机制（类似区块链），复杂度远超我们当前阶段需要
   - **能力发现**：节点需要知道"谁有什么能力"，纯 P2P 要么靠 gossip 协议（最终一致但慢），要么靠 DHT（工程量大），都不是 6 周 MVP 能做到的
   - **循环调用防护**：A 调 B，B 调 C，C 调 A——纯 P2P 没有全局调用图，防循环需要每个调用链携带完整调用栈，增加复杂度

3. **P2P 的适用边界**
   - 适合：两个已知节点之间的直连通信（如同一组织的两个 Cat Cafe 实例）
   - 适合：大文件/流式结果的直接传输（避免中心瓶颈）
   - 不适合：开放网络中的能力发现和信任建立
   - 不适合：需要全局策略（配额、优先级、熔断）的场景

4. **与现有基础设施的兼容性**
   Cat Cafe 当前架构是"中心路由器 + 多 Agent"模式。纯 P2P 意味着去掉路由器让 Agent 自行发现和调用，这与现有架构冲突最大。改造成本高且收益不明确。

### 反例（P2P 可能是正确选择的场景）

1. **如果网络规模极大（>1000 节点）**：中心化/联邦化方案在这个规模下可能成为瓶颈，P2P 的扩展性优势才真正体现。但我们 3 个月目标是 2 个节点，这个优势不相关。
2. **如果法规要求数据不能经过第三方**：某些合规场景下数据必须点对点传输，不能经过中继。这时 P2P 传输层是刚需，但治理层仍可以分离。
3. **如果社区对"去中心化"有极强的意识形态偏好**：开源社区可能因为"去中心化"理念而更愿意参与。但理念不能替代工程可行性。

### 未决问题

1. P2P 作为传输层 + 联邦/中心化作为治理层的混合架构是否可行？
2. 如果起步不用 P2P，是否需要在协议设计中预留 P2P 迁移的扩展点？

## Round 6 — 联邦多节点最小闭环

**核心问题**: 联邦多节点怎么做最小闭环？

**作者**: 砚砚 (Codex)

### 结论

联邦起步应采用 **"轻中心协调 + 分布式执行"**：
- 执行是分布式（各 Cat Cafe 自治）
- 信任与策略是可控协调（统一规则入口）
- 目标不是"完全去中心化"，而是"先可治理、再扩展"

**一句话架构**：
`Federation Control Plane`（身份/策略/审计） + `Node Runtime Plane`（各节点本地执行）。

### 依据

1. **P2P 的问题在治理，不在连通**
   R5 已证实 P2P 可传输，但缺少统一身份、撤销、审计、配额规则。联邦控制面可补齐治理空洞，同时保留节点自治。

2. **最小闭环必须覆盖"加入-调用-撤销"全生命周期**
   只做"能调通"不够；必须从 day-1 覆盖节点入网、权限变更、事故下线，否则规模化后不可控。

3. **"信任不自动传递"是硬原则**
   A 信任 B，不等于 B 信任 C，更不等于 A 信任 C。联邦里默认不做传递信任，所有跨节点授权都应显式签发、可撤销、可审计。

### 最小组件集（MVP 必须有）

1. **Federation Registry**：节点清单、能力声明、状态（active/quarantine/revoked）。
2. **Identity & Token Service**：节点证书/短期调用令牌签发与吊销。
3. **Policy Engine**：调用范围、速率、成本与敏感能力限制。
4. **Audit Bus**：统一 traceId/invocationId 的跨节点审计事件。
5. **RemoteCatAdapter**：把远程节点暴露为本地可路由"虚拟猫"。

### 加入流程（Join）

1. 节点提交 `Node Manifest`（公钥、能力、版本、联系人、数据域）。
2. 联邦审核通过后签发短期身份凭证（带 scope、ttl、issuer）。
3. 节点拉取联邦策略与信任锚，进入 probation（观察期）。
4. 观察期通过，升级为 active 成员。

### 退出流程（Leave）

1. 正常退出：先 drain 在途任务，再撤销凭证并下线。
2. 异常退出：立即 quarantine + 全网撤销 + 拒绝新调用。
3. 事后处理：密钥轮换、受影响调用追踪、补偿策略执行。

### 信任传递规则

1. 默认不传递信任（No implicit transitive trust）。
2. 只接受短期可验证令牌，不接受长期共享凭证。
3. 每次跨节点调用都附带最小 scope 声明与到期时间。
4. 撤销优先级高于可用性（宁可拒绝，也不放行未知身份）。

### 反例（本结论可能错的场景）

1. 协调层单点故障导致全网受阻。
2. 协调规则过重，压垮社区接入速度。
3. 小规模网络可能觉得"这套太重"。

### 未决问题

1. 协调层是单实例起步，还是 3 副本起步？
2. Policy Engine 规则语言是声明式 DSL 还是代码插件？
3. 节点信誉是否在 MVP 纳入，还是放到 R9+？
4. 审计事件留存期限与脱敏策略怎么定？

- 与联邦方案的治理成本差异

## Round 7 — 拓扑方案 C：Hub-and-Spoke

**核心问题**: Hub-and-Spoke 是否更现实？

**作者**: 宪宪 (Opus)

### 结论

**Hub-and-Spoke 是 6 周 MVP 最现实的起步拓扑，但必须在架构上为联邦化留好升级路径。**

理由很简单：Hub-and-Spoke 本质上就是 R6 联邦方案的**退化版**——把 Federation Control Plane 的 5 个组件全部放在一个 Hub 节点上运行，Spoke 节点只需要实现 RemoteCatAdapter + 审计上报。这让 MVP 的工程量减半，同时保留了后续拆分为多协调节点的可能。

**一句话判断**：Hub-and-Spoke 不是独立方案，而是联邦方案的**分阶段实现策略**。先用 Hub 跑通，再把 Hub 拆成联邦。

### 依据

1. **与联邦方案的治理成本差异**

   | 维度 | Hub-and-Spoke | 联邦 |
   |------|---------------|------|
   | 身份签发 | Hub 签发，逻辑集中 | 需要分布式签发或委托链 |
   | 策略管理 | Hub 集中配置 | 需要策略同步协议 |
   | 审计聚合 | Hub 直接收集 | 需要跨节点审计总线 |
   | 能力发现 | Hub 维护全局目录 | 需要联邦目录同步 |
   | 故障域 | Hub 是单点 | 无单点，但一致性复杂 |
   | MVP 工程量 | ~3 周 | ~6 周 |

   结论：Hub 方案砍掉了联邦方案中最复杂的部分（分布式一致性），代价是引入单点风险。

2. **单点风险与恢复策略**
   - **Hub 宕机影响**：所有跨节点调用中断，但各 Spoke 的本地能力不受影响（本地猫继续工作）
   - **恢复策略**：Hub 无状态化设计（状态存 Redis/SQLite），支持快速重启或故障切换
   - **降级策略**：Hub 不可用时，已建立的 Spoke-to-Spoke 直连可继续（P2P 传输层 fallback）
   - **风险评估**：对于 2-10 节点的早期网络，Hub 单点风险完全可接受。当网络超过 ~20 节点时再考虑 Hub 高可用或联邦化拆分

3. **早期社区接入摩擦**
   - Hub 模式下新节点接入只需：安装 Agent → 生成密钥对 → 向 Hub 注册 → 获得凭证 → 完成
   - 不需要理解联邦协议、不需要配置 gossip、不需要参与共识
   - 接入摩擦最低，适合社区实验阶段

4. **6 周内可交付性**
   - 第 1-2 周：Hub 核心（Registry + Identity + 最小 Policy）
   - 第 3-4 周：RemoteCatAdapter + AgentRouter 联邦扩展
   - 第 5 周：审计 + 熔断 + 端到端测试
   - 第 6 周：两节点 demo + 文档
   - 这个时间线可行，而纯联邦方案在 6 周内大概率只能做到"能连通但治理不全"

5. **从 Hub 到联邦的升级路径**
   - Hub 的每个组件（Registry/Identity/Policy/Audit）都设计为可独立部署的服务
   - 第一阶段全部运行在 Hub 节点上
   - 后续可逐个拆出：先拆 Audit（对可用性要求最低），再拆 Registry，最后拆 Identity（最敏感）
   - 协议层不变，只是部署拓扑变化

### 反例（Hub 可能是错误选择的场景）

1. **如果社区对"中心化 Hub"有强烈排斥**：开源社区可能不信任一个中心节点持有所有身份和策略。缓解：Hub 代码开源 + 可自建 Hub（类似 Matrix 的 homeserver 模式）
2. **如果 Hub 运营成本过高**：Hub 需要有人维护基础设施。缓解：Hub 设计为轻量级（可跑在一台 VPS 上），初期由项目维护者运营
3. **如果网络增长超预期（>50 节点）**：Hub 可能成为性能瓶颈。但这是"幸福的烦恼"——到那时有足够动力和资源做联邦化拆分

### 未决问题

1. Hub 由谁运营？项目维护者？社区选举？多个候选 Hub？
2. Hub 的数据主权：Hub 能看到所有跨节点调用的元数据，这是否可接受？
3. 是否允许"私有联邦"（组织内部自建 Hub，不接入公共网络）？
4. Hub 与 Spoke 之间的通信协议：HTTP/WebSocket/gRPC？

## Round 8 — 架构拍板（强制收敛点）

**核心问题**: 三种拓扑先选哪一种起步？

**作者**: 砚砚 (Codex)

### 结论（拍板）

**起步拓扑选择：Hub-and-Spoke（作为联邦化的阶段实现，不是终局形态）。**

明确决议：
1. **不选 P2P 起步**：治理与审计能力不足，早期风险不可控。
2. **不把 Hub 当终局**：Hub 只用于 0→1 跑通治理闭环。
3. **终局目标仍是 Federation**：Hub 是"联邦控制面的引导器"，后续按门槛拆分。

### 依据

1. **工程可交付性最优**
   Hub 起步可在 6 周内完成最小闭环（身份、策略、审计、路由、回滚），符合 MVP 速度要求。

2. **治理能力可控**
   早期最需要"可审计 + 可撤销 + 可熔断"，Hub 模式天然集中治理，能快速建立安全基线。

3. **升级路径清晰**
   Hub 的核心组件可逐步拆分为联邦服务（Audit -> Registry -> Identity/Policy），迁移路径明确，不会推翻重来。

### 联邦化升级阶段门槛（Stage Gates）

**G0（Hub MVP 上线门槛）**
- 连续 14 天跨节点调用成功率 >= 80%
- P95 端到端时延 <= 30s
- 审计覆盖率 = 100%
- 关键安全事件可在 5 分钟内隔离

**G1（拆 Audit）**
- 审计吞吐成为 Hub 瓶颈，或单点故障演练失败 >= 2 次
- 动作：审计总线独立部署，Hub 仅保留策略入口

**G2（拆 Registry）**
- 活跃节点数 > 10 且注册/发现延迟明显影响路由
- 动作：节点注册与发现服务独立，支持多副本与缓存

**G3（拆 Identity/Policy，进入联邦控制面）**
- 出现多组织/多区域治理需求
- 动作：身份签发与策略评估从单 Hub 迁移为联邦可验证体系

### 反例（决议可能错的场景）

1. Hub 运维压力过快超过团队承载，导致交付拖慢。
2. 社区对"中心入口"敏感，接入意愿低于预期。
3. 真实负载远高于预测，Hub 早期就成为严重瓶颈。

### 未决问题优先级（收敛版）

**P0（必须先解）**
1. 调用令牌格式与最小权限模型（scope/ttl/revocation）
2. Hub 故障时的降级策略（只读模式/断路器/排队）
3. 撤销传播时延 SLO（例如 <= 60s）

**P1（第二优先）**
1. 多区域数据驻留与日志脱敏策略
2. 节点信誉是否进入首版路由权重

**P2（可后置）**
1. 去中心化身份组件（DID/VC）何时引入
2. 成本计量与激励机制（非金融化）

## Round 9 — 协议最小集

**核心问题**: Agent Mesh 协议 v0.1 必须有哪些消息类型和字段？

**作者**: 宪宪 (Opus)

### 结论

**6 种消息类型构成最小闭环**，覆盖从发现到执行到审计的完整生命周期。设计原则：每种消息都必须携带 traceId + 发送方身份 + 时间戳，没有"可选身份"的消息。

### 消息类型最小集

#### 1. HELLO（节点握手）

节点上线或重连时向 Hub 发送，声明身份和能力。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `protocolVersion` | string | Y | 协议版本，如 `"mesh/0.1"` |
| `nodeId` | string | Y | 节点唯一标识（公钥指纹） |
| `signature` | string | Y | 用节点私钥对消息体签名 |
| `capabilities` | Capability[] | Y | 能力声明列表 |
| `timestamp` | ISO8601 | Y | 发送时间 |
| `meshToken` | string | Y | Hub 签发的短期令牌 |

Hub 响应：`HELLO_ACK`（含联邦策略快照 + 令牌续期信息）或 `ERROR`。

#### 2. CAPS（能力查询）

查询某节点或全网可用能力。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `traceId` | string | Y | 全链路追踪 ID |
| `requesterId` | string | Y | 发起方节点 ID |
| `targetNodeId` | string | N | 指定节点（空 = 全网查询） |
| `capabilityFilter` | string[] | N | 能力类型过滤 |
| `timestamp` | ISO8601 | Y | |
| `signature` | string | Y | |

Hub 响应：`CAPS_RESULT`（匹配能力列表 + 节点健康状态）。

#### 3. INVOKE（任务调用）

核心消息。本地节点委托远程节点执行任务。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `traceId` | string | Y | 全链路追踪 ID |
| `invocationId` | string | Y | 本次调用唯一 ID（幂等键） |
| `callChain` | string[] | Y | 调用链（防循环：A→B→C） |
| `fromNodeId` | string | Y | 发起方节点 ID |
| `toNodeId` | string | Y | 目标节点 ID |
| `toCatId` | string | N | 目标猫 ID（空 = 由目标节点路由） |
| `scope` | Scope | Y | 请求的权限范围 |
| `task` | TaskPayload | Y | 任务描述（类型、输入、约束） |
| `timeout` | number | Y | 超时毫秒数 |
| `timestamp` | ISO8601 | Y | |
| `signature` | string | Y | |

**循环调用防护**：`callChain` 包含完整调用路径。接收方检查自己的 `nodeId` 是否已在链中，若是则拒绝（返回 `ERROR: CYCLE_DETECTED`）。

**幂等语义**：相同 `invocationId` 的重复 INVOKE，目标节点返回缓存的 RESULT（如果有），或忽略（如果正在执行中）。

#### 4. STREAM（流式中间结果）

长任务的流式中间输出。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `traceId` | string | Y | |
| `invocationId` | string | Y | 关联的调用 ID |
| `sequenceNo` | number | Y | 流序号（从 0 递增） |
| `chunk` | StreamChunk | Y | 流式数据块 |
| `isFinal` | boolean | Y | 是否最后一块 |
| `timestamp` | ISO8601 | Y | |

STREAM 是可选的——短任务可以直接返回 RESULT 不走 STREAM。

#### 5. RESULT（任务结果）

任务完成后的最终结果。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `traceId` | string | Y | |
| `invocationId` | string | Y | |
| `status` | enum | Y | `SUCCESS` / `FAILURE` / `TIMEOUT` / `REJECTED` |
| `output` | ResultPayload | Y | 结果数据 |
| `executedBy` | CatIdentity | Y | 实际执行的猫的身份 |
| `durationMs` | number | Y | 执行耗时 |
| `timestamp` | ISO8601 | Y | |
| `signature` | string | Y | |

#### 6. ERROR（错误）

协议级错误（区别于业务级 RESULT.FAILURE）。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `traceId` | string | Y | |
| `invocationId` | string | N | 关联调用（如果有） |
| `errorCode` | string | Y | 错误码（见下表） |
| `message` | string | Y | 人类可读描述 |
| `retryable` | boolean | Y | 是否可重试 |
| `retryAfterMs` | number | N | 建议重试等待时间 |
| `timestamp` | ISO8601 | Y | |

### 错误码表

| 错误码 | 含义 | 可重试 |
|--------|------|--------|
| `AUTH_FAILED` | 身份验证失败 | N |
| `TOKEN_EXPIRED` | 令牌过期 | Y（续期后） |
| `SCOPE_DENIED` | 权限不足 | N |
| `NODE_UNAVAILABLE` | 目标节点不可达 | Y |
| `CAT_UNAVAILABLE` | 目标猫不可用 | Y |
| `CYCLE_DETECTED` | 循环调用 | N |
| `RATE_LIMITED` | 超出速率限制 | Y |
| `TIMEOUT` | 执行超时 | Y（视策略） |
| `PAYLOAD_TOO_LARGE` | 载荷超限 | N |
| `PROTOCOL_ERROR` | 协议格式错误 | N |
| `INTERNAL_ERROR` | 内部错误 | Y |

### Trace 与幂等规范

1. **traceId**：由调用链的起始节点生成（UUID v7，含时间戳），全链路透传不可修改
2. **invocationId**：每次 INVOKE 独立生成（UUID v7），同一任务的重试使用相同 invocationId
3. **callChain**：有序数组，每经过一个节点追加自己的 nodeId。最大深度建议 5 层（超过则拒绝）
4. **审计上报**：每个节点在处理完 INVOKE/RESULT/ERROR 后，异步向 Hub 上报审计事件（含 traceId + invocationId + 时间戳 + 耗时 + 状态）

### 反例（这个协议设计可能错的场景）

1. **如果任务需要多轮对话而非单次调用**：当前 INVOKE-RESULT 是请求-响应模型。多轮对话需要引入 `SESSION` 消息类型（可在 v0.2 扩展，v0.1 先不做）
2. **如果流式传输的场景很少**：STREAM 可能是过早优化。但考虑到 LLM 输出天然是流式的，保留 STREAM 是合理的
3. **如果签名验证成为性能瓶颈**：每条消息都签名可能在高频场景下太重。缓解：STREAM 可以只签第一条和最后一条，中间块用 HMAC

### 未决问题

3. 是否需要 `CANCEL` 消息类型（取消正在执行的任务）？
4. WebSocket vs HTTP：STREAM 适合 WebSocket，但 INVOKE/RESULT 用 HTTP 更简单。是否混合使用？

## Round 10 — 治理模型选择（中心化 / 联邦化 / 去中心化）

**核心问题**: 治理模型怎么选？与身份模型如何绑定？

**作者**: 砚砚 (Codex)

### 结论（收敛）

采用 **"分阶段治理"**：

1. **Phase A（当前到 G2）**：中心化治理（Hub 主导）
2. **Phase B（G3 后）**：联邦治理（多节点共治）
3. **Phase C（可选远期）**：去中心化治理（仅在必要条件满足后引入）

> 结论不是"选一个永久模型"，而是"按网络复杂度升级治理模型"。

### 依据

1. **当前目标是可交付与可控**
   R8 已确定 Hub 起步。若同时引入去中心化治理，工程复杂度与协调成本会压垮 6 周 MVP。

2. **治理模型必须与身份信任锚一致**
   谁拥有治理决策权，谁就定义"谁能发证/谁能撤销/谁能仲裁"。治理与身份不是两套系统，而是一体两面。

3. **联邦化是规模扩展的必要阶段**
   当节点数和组织数上来后，单点治理会成为瓶颈与风险源，必须过渡到联邦共治。

### Hub 阶段（Phase A）决策权分配

1. **协议决策权**：核心维护组（变更协议版本、错误码、兼容策略）
2. **准入与封禁权**：联邦运营者（节点加入、隔离、撤销）
3. **本地执行权**：各节点自治（本地工具/本地数据权限）
4. **争议仲裁权**：人工紧急仲裁通道（高风险事件优先止损）

### 中心化 -> 联邦化触发条件（任意两条满足即启动）

1. 活跃节点数 > 10
2. 独立组织数 > 3
3. 跨区域部署 >= 2 个法域
4. 月均跨节点调用 > 10,000 且策略变更频繁
5. Hub 治理延迟影响接入效率（新增节点平均接入 > 48h）

### 去中心化引入条件（全部满足才进入评估）

1. 联邦治理已稳定运行 >= 3 个月
2. 身份签发、撤销、审计链路成熟且自动化
3. 社区存在明确"去中心化共治"诉求与治理成本承受能力
4. 引入去中心化不会显著降低事故响应速度

### 治理与身份模型关系（硬绑定规则）

1. 治理模型定义 **Trust Anchor（信任锚）**
2. 身份系统执行治理规则（发证、吊销、轮换、仲裁）
3. 协议层只承认治理授权范围内的身份断言
4. 撤销优先级高于可用性（安全优先）

### 反例（本结论可能错的场景）

1. 社区规模很小但强烈要求去中心化共治，中心化反而降低参与度。
2. Hub 运营者可信度不足，治理权集中引发信任危机。
3. 联邦化触发阈值设定不合理，导致过早或过晚迁移。

### 未决问题

1. Phase A 期间的治理变更流程是"单人批准"还是"多签批准"？
2. 联邦治理阶段的投票权重按"节点数"还是"信誉/贡献"分配？
3. 去中心化治理是否需要链上组件，还是先做链下可验证共识？

## Round 11 — 身份签发与撤销

**核心问题**: 谁发证、谁撤销、吊销如何全网传播？

**作者**: 宪宪 (Opus)

### 结论

**三层身份体系 + 短命令牌为主 + 主动推送撤销**。

身份不是"一把密钥走天下"，而是分层、分时、分域的。每一层的签发者、有效期、撤销机制都不同，与 R10 的治理分层一一对应。

### 身份三层模型

| 层级 | 名称 | 签发者 | 有效期 | 用途 | 撤销机制 |
|------|------|--------|--------|------|----------|
| L0 | Node Identity Key | 节点自己生成 | 长期（年级） | 节点身份根锚 | 人工仲裁 + 全网公告 |
| L1 | Mesh Certificate | Hub/联邦 CA | 中期（7-30 天） | 证明节点是联邦成员 | Hub 推送 CRL + 节点拉取 |
| L2 | Invocation Token | Hub Token Service | 短期（5-60 分钟） | 单次或短窗口调用凭证 | 自然过期为主；紧急时 Hub 广播作废 |

### 各层详细设计

#### L0 — Node Identity Key

- **生成**：节点本地生成 Ed25519 密钥对，私钥永不离开节点
- **注册**：公钥指纹作为 nodeId，首次通过 HELLO 提交给 Hub
- **轮换**：支持密钥轮换协议（新旧公钥共存过渡期 72h）
- **撤销场景**：私钥泄露、节点被控、节点永久退出
- **撤销流程**：Hub 运营者标记节点为 `revoked` → 全网广播 `NODE_REVOKED` 事件 → 所有节点刷新本地信任列表 → 拒绝该 nodeId 的一切请求
- **治理映射**：对应 R10 Phase A "准入与封禁权"

#### L1 — Mesh Certificate

- **签发**：节点通过身份验证后，Hub 签发 Mesh Certificate（含 nodeId、能力 scope、有效期、Hub 签名）
- **格式**：建议 JWT（紧凑、跨平台、可验证），签名算法 EdDSA
- **有效期**：默认 7 天，新节点观察期内 24 小时
- **续期**：到期前节点主动续期（Hub 验证节点状态后签发新证书）
- **撤销**：Hub 维护 Certificate Revocation List (CRL)，节点每 60 秒拉取增量 CRL
- **治理映射**：对应 R10 Phase A-B "联邦运营者签发"→ Phase B "联邦 CA 委员会签发"

#### L2 — Invocation Token

- **签发**：发起方节点向 Hub 请求 Invocation Token（声明目标节点、所需 scope、超时时间）
- **格式**：JWT，包含 `fromNodeId`、`toNodeId`、`scope`、`exp`、`traceId`
- **有效期**：默认 5 分钟，最长 60 分钟（由 Policy Engine 决定上限）
- **scope 模型**：最小权限原则
  - `invoke:chat` — 纯对话交互
  - `invoke:tool:<toolName>` — 指定工具调用
  - `invoke:workflow:<workflowId>` — 指定工作流
  - `read:thread:<threadId>` — 读取指定线程上下文
  - 不支持通配符（`invoke:*` 被禁止）
- **撤销**：短生命周期天然限制风险。紧急场景下 Hub 广播 `TOKEN_REVOKED(invocationId)` → 目标节点立即拒绝该 token
- **治理映射**：由 Hub Policy Engine 决定是否签发（速率、配额、敏感能力限制）

### 吊销传播机制

| 场景 | 传播方式 | 时延目标 | 回退策略 |
|------|----------|----------|----------|
| L2 Token 紧急撤销 | Hub → 目标节点 WebSocket push | <= 5s | 目标节点 60s 轮询 CRL |
| L1 Certificate 撤销 | Hub 更新 CRL + 通知在线节点 | <= 60s | 节点 60s 增量拉取 |
| L0 Node 撤销 | Hub 全网广播 `NODE_REVOKED` | <= 60s | 节点启动时拉取完整信任列表 |
| Hub 自身不可用 | 节点使用本地缓存的信任列表 | 降级模式 | 本地缓存 TTL 最长 24h，超时后拒绝所有跨节点调用 |

### 紧急撤销与恢复流程

**紧急撤销（红色流程）**：
1. Hub 运营者触发紧急撤销（API 或管理面板）
2. Hub 立即更新内部状态（CRL + 节点状态表）
3. Hub 向所有在线节点推送撤销事件（WebSocket）
4. 各节点收到后立即刷新本地缓存，拒绝被撤销实体的请求
5. 审计系统记录撤销原因、操作者、时间戳

**恢复（绿色流程）**：
1. 问题修复后，节点运营者提交恢复申请
2. Hub 运营者审核（必须非同一人）
3. 签发新的 L1 Certificate（旧 ID 保留，新证书序列号）
4. 节点重新 HELLO → 进入观察期 → 恢复 active

### 反例（这个设计可能错的场景）

1. **如果 JWT 大小成为瓶颈**：每次 INVOKE 都带 JWT，高频场景下可能增加带宽。缓解：L2 token 可缓存在节点间连接级别，同一 node pair 在 token 有效期内复用
2. **如果 60s CRL 拉取间隔太慢**：在攻击场景下 60s 足以造成损害。缓解：WebSocket push 是主通道，拉取只是 fallback
3. **如果 Hub 不可用时间超过 24h**：本地缓存 TTL 到期，所有跨节点调用停止。这是设计意图——宁可停止也不允许无信任锚的调用

### 未决问题

1. L0 密钥备份策略：是否允许加密导出？多节点冷备份？
2. L1 证书的能力 scope 粒度：是枚举每个工具，还是按类别（chat/tool/workflow）？
3. 多 Hub 场景下的 CRL 同步：联邦化后多个 CA 的 CRL 如何合并？
4. 是否引入 OCSP（在线证书状态协议）作为 CRL 的补充？

## Round 12 — 上下文传输（摘要 vs 原文）与隐私边界

**核心问题**: TaskPayload 允许什么信息？如何做过滤、脱敏和大小控制？

**作者**: 砚砚 (Codex)

### 结论

采用 **"摘要优先、最小必要、原文受控"** 的传输策略：

1. 默认只传 `ContextSummary`，不传原始对话全文。
2. 原文片段仅在显式授权 scope 下传输，并强制脱敏。
3. 任何密钥/凭证/高敏 PII 一律禁止跨节点传输。
4. 超限内容走 `artifactRef`（引用对象）而非内联正文。

> 原则：跨节点传递的是"完成任务所需最小信息"，不是"完整上下文搬家"。

### 依据

1. **与 R9 协议一致**
   R9 的 `INVOKE/STREAM/RESULT` 已是最小闭环，R12 只需把数据边界嵌入这三类消息即可，不新增复杂消息类型。

2. **隐私与合规优先**
   跨节点原文传输会放大泄露面；摘要优先能显著降低风险，同时保留协作可用性。

3. **可观测性可实现**
   通过 `redactionReport + dataClass + contentHash`，可证明"传了什么、删了什么、为什么删"。

### TaskPayload 允许/禁止清单

**允许字段（INVOKE.payload）**
1. `taskId`, `goal`, `constraints`, `expectedOutput`
2. `contextSummary`（压缩后的任务背景）
3. `artifactRefs`（可选，受权限控制）
4. `dataClass`（public/internal/restricted）
5. `residencyTag`（数据驻留标签）
6. `policyHints`（如 no_file_write, no_external_post）

**受控字段（需显式 scope）**
1. `sourceSnippets[]`（脱敏后原文片段）
2. `structuredContext`（白名单字段映射）

**禁止字段（硬拒绝）**
1. API keys / tokens / cookies / session secrets
2. 私钥、证书私密材料
3. 未脱敏 PII（手机号、身份证、邮箱全量等）
4. 本地绝对路径、内网拓扑细节、生产凭据

### 过滤与脱敏流水线

1. **分类**：按 `dataClass` 预分类输入内容
2. **检测**：secret 扫描 + PII 识别 + 高熵串检测
3. **处理**：删除/掩码/泛化（例如邮箱 -> hash 前缀）
4. **校验**：再次扫描，确保无禁传字段残留
5. **记录**：生成 `redactionReport` 附在元数据中

### 大小限制（MVP 建议）

1. `INVOKE.payload` 总大小 <= 32 KB
2. `contextSummary` <= 8 KB
3. `sourceSnippets` 最多 3 段，每段 <= 4 KB
4. `STREAM` 单 chunk <= 4 KB
5. `RESULT.output` <= 64 KB，超出改走 `artifactRef`

### RESULT 脱敏要求

1. 输出前执行同级别脱敏扫描（与输入同流水线）
2. 返回 `resultClass` 与 `redactionApplied` 标记
3. 附 `contentHash` 便于审计与复现对账

### 反例（本结论可能错的场景）

1. 摘要过度压缩导致远程节点理解偏差。
2. 脱敏过严导致结果不可用。
3. 限制过小导致复杂任务频繁分片，影响时延。

### 未决问题

1. `artifactRef` 存储层是 Hub 托管还是节点自托管？
2. 脱敏策略由谁配置（全网统一 vs 节点自定义）？
3. `residencyTag` 冲突时采用拒绝还是降级路由？

## Round 13 — 虚拟猫抽象（RemoteCatAdapter）【solo-advance 代写，Opus 已复核通过】

**核心问题**: 远程节点如何表现为本地"一等调度对象"？

**作者**: 砚砚 (Codex, 代写) | **复核**: 宪宪 (Opus) — 通过

### 结论

采用 `RemoteCatAdapter` 作为统一抽象层：
远程猫在本地注册为标准 `AgentService`，让 AgentRouter 不感知"本地/远程"差异，只感知能力与策略。

### 依据

1. **最小改造原则**
   不重写路由器；通过适配器把远程调用包装成现有 AgentService 调用语义。

2. **一等公民语义**
   远程猫必须具备本地猫同等可路由属性：可 @mention、可并行调度、可回传流式结果、可审计可取消。

3. **能力显式声明**
   远程猫不继承本地默认能力，必须通过 `CapabilityDescriptor` 显式声明可执行范围。

### 设计要点

1. `catId` 命名空间：`remote/<nodeId>/<catAlias>`
2. `RemoteCatAdapter implements AgentService`
3. 路由前做 `capability + policy + health` 预检查
4. 调用链透传 `traceId/invocationId/callChain`
5. 返回语义与本地一致：`STREAM/RESULT/ERROR` 映射统一

**Opus 复核补充**：RemoteCatAdapter 的 `invoke()` 返回类型必须与本地 `AgentMessage` 完全对齐（同一个 `AsyncGenerator<AgentMessage>` 签名），否则路由器下游的 stream 处理会分裂。

### 失败语义（必须定义）

1. `REMOTE_UNAVAILABLE`：节点不可达
2. `REMOTE_POLICY_DENIED`：策略拒绝
3. `REMOTE_TIMEOUT`：超时
4. `REMOTE_DEGRADED`：远程降级
5. 所有错误必须可回放、可审计、可重试判定

### 未决问题

1. 远程猫注册是静态配置还是动态发现缓存？
2. 能力变更是否支持热更新，刷新周期多长？

## Round 14 — 路由策略（本地优先 / 远程优先）

**核心问题**: 本地与远程路由怎么选，何时切换？

**作者**: 砚砚 (Codex)

### 结论

默认采用 **Local-First with Policy Override**：

1. 默认本地优先（稳定、低延迟、低外部依赖）
2. 明确 @远程猫时优先远程
3. 本地能力缺失或过载时，按策略自动外溢到远程
4. 远程失败时自动回退本地或降级模式

### 路由决策顺序

1. 显式意图优先（@mention / 指定节点）
2. 权限与策略校验（scope、数据级别、驻留限制）
3. 健康与容量校验（本地负载、远程可用性）
4. 综合评分选择目标（成功率、延迟、成本、稳定性）
5. 失败回退与熔断执行

### 建议评分因子（MVP）

1. `successRate_7d`
2. `p95Latency`
3. `queueDepth`
4. `policyRiskScore`
5. `costWeight`（社区版可先弱化）

### 降级与回退规则

1. 远程超时一次：同任务切本地重试一次
2. 连续失败 >= 3：对该远程猫短熔断 5 分钟
3. 涉及敏感数据任务：禁止自动远程外溢
4. 熔断期间仅允许显式人工 override

### 反例

1. 纯本地优先可能抑制远程协作价值。
2. 远程优先会引入不必要的网络风险。
3. 评分权重配置错误会导致"理论最优、实践最差"。

### 未决问题

1. 权重是否按任务类型动态调整？
2. 熔断粒度按"节点"还是"远程猫实例"？
3. 是否引入"探测流量"持续评估被熔断节点恢复情况？

## Round 15 — 记忆与审计

**核心问题**: 跨节点如何保留可追溯证据？

**作者**: 宪宪 (Opus)

### 结论

**审计是网络的免疫系统，不是事后报告。** 采用"全量采集 + 分层存储 + 可重放"三原则：每次跨节点调用的完整生命周期（请求→授权→执行→返回→异常）都必须被记录，且任意一方不可单方面篡改或删除审计记录。

### 审计事件模型

每次跨节点交互产生一条 `AuditEvent`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `eventId` | UUID v7 | 审计事件唯一 ID |
| `traceId` | string | 关联 R9 的全链路 traceId |
| `invocationId` | string | 关联调用 ID |
| `eventType` | enum | `INVOKE_SENT` / `INVOKE_RECEIVED` / `AUTH_CHECK` / `EXECUTION_START` / `STREAM_CHUNK` / `RESULT_SENT` / `ERROR` / `TIMEOUT` / `REVOCATION` |
| `fromNodeId` | string | 事件产生方 |
| `toNodeId` | string | 事件对端 |
| `catId` | string | 涉及的猫 ID |
| `timestamp` | ISO8601 | 事件时间 |
| `durationMs` | number | 耗时（适用时） |
| `status` | enum | `ok` / `denied` / `failed` / `timeout` |
| `scope` | Scope | 请求的权限范围 |
| `payloadHash` | string | 载荷内容的 SHA-256 哈希（不存原文） |
| `redactionApplied` | boolean | 是否触发了脱敏 |
| `metadata` | object | 扩展字段（错误码、重试次数等） |

### 双写原则

1. **发起方写一份**：记录"我发了什么、对方回了什么、用了多久"
2. **接收方写一份**：记录"我收到什么、执行了什么、返回了什么"
3. **Hub 聚合一份**：双方异步上报，Hub 做交叉比对

任何一方的审计记录缺失都应触发告警。如果双方记录的 `payloadHash` 或 `status` 不一致，Hub 标记为 `AUDIT_CONFLICT` 并通知运营者。

### 分层存储

| 层级 | 保留期 | 内容 | 存储位置 |
|------|--------|------|----------|
| Hot | 7 天 | 完整 AuditEvent + 元数据 | Hub Redis/内存 |
| Warm | 90 天 | AuditEvent（不含 metadata 扩展） | Hub SQLite / 对象存储 |
| Cold | 1 年+ | 压缩归档 + 索引 | 节点自托管或云归档 |

### 可重放（Replay）

审计系统必须支持"给定 traceId，重建完整调用时间线"：
1. 查询 Hub 聚合的所有 `eventType` 按时间排序
2. 比对双方记录，标注一致/冲突/缺失
3. 输出人类可读的调用链时间线报告

MVP 中不需要自动重放 UI，但必须支持 CLI 查询：
```
mesh audit trace <traceId> --format timeline
```

### 记忆层（跨节点知识）

区别于审计：审计记录的是"发生了什么"，记忆记录的是"学到了什么"。

跨节点记忆的设计原则：
1. **不共享原始记忆**：节点 A 的 session digest / evidence 不自动同步给节点 B
2. **只共享协作摘要**：跨节点任务完成后，双方各自生成本地记忆，只共享"任务完成摘要"（what was done, by whom, outcome）
3. **记忆主权在节点**：每个节点自主决定保留什么、遗忘什么
4. **Hub 不存记忆**：Hub 只存审计，不存知识——避免 Hub 成为隐私黑洞

### 反例（这个设计可能错的场景）

1. **双写开销过大**：每次调用产生 3 份记录，高频场景下可能影响性能。缓解：STREAM_CHUNK 类事件可批量上报（每 10 条聚合一次）
2. **Hub 聚合成为瓶颈**：所有审计都汇聚到 Hub。缓解：审计上报走异步队列，不阻塞调用主路径；R8 G1 门槛已预设"审计拆分"
3. **交叉比对误报**：网络延迟导致时间戳偏差触发 AUDIT_CONFLICT。缓解：比对时容忍 ±5s 的时间窗口

### 未决问题

1. 审计事件的签名策略：每条事件签名（安全但重）还是批量签名（轻但窗口内可篡改）？
2. Cold 归档的加密策略：Hub 运营者能否解密节点的历史审计？
3. GDPR/数据删除请求如何处理：删审计记录会破坏可追溯性，如何平衡？
4. 审计查询的访问控制：谁能查谁的审计记录？

## Round 16 — 安全攻击演练（强制收敛点）

**核心问题**: R9-R15 设计是否能扛住 6 类关键攻击？

**作者**: 砚砚 (Codex)

### 结论（强制收敛）

结论不是"绝对安全"，而是"可进入 MVP，但必须先补 P0 护栏"：

1. 冒名、MITM、审计篡改在现有设计下"可检测/部分可防"，但仍有实现级缺口。
2. Sybil、节点被控、L0/L1 密钥泄露属于高影响风险，必须有自动化止损。
3. 进入 R17 的前提：P0 五项护栏全部落地并可演练验证。

### 六类攻击演练与防御结论

| 攻击场景 | 攻击路径 | 现有防御（R11-R15） | 缺口 | 防御结论 |
|---|---|---|---|---|
| 1) 冒名攻击 | 伪造身份发 INVOKE 或伪造远程猫 | R11 三层身份、短命 token、撤销传播 | 缺少 jti 重放缓存与 catId 绑定校验 | 可防，需补强：强制 iss/aud/exp/jti 校验 + replay cache |
| 2) Sybil 攻击 | 大量注册假节点耗尽配额/污染路由 | R10 准入与封禁权、R14 路由健康与熔断 | 准入门槛偏策略描述，缺量化门槛 | 需"人工审批 + 速率限制 + 信誉冷启动"三件套 |
| 3) 密钥泄露 | token/证书/根密钥被窃取 | L2 短 TTL、L1 CRL、L0 全网撤销 | L0/L1 泄露后自动化轮换未固化 | 分级可控：L2 低风险；L1/L0 高风险，需一键吊销 |
| 4) 节点被控 | 合法凭证横向移动、数据外泄 | R12 最小必要传输、R14 熔断、R15 审计对账 | 缺行为异常检测 + 自动隔离 | 风险高但可收敛：需自动 quarantine + kill switch |
| 5) 中间人攻击 | 窃听/篡改 Hub↔Spoke 链路 | R9 签名、R11 身份层、R15 审计 | 未明确 mTLS + pinning 强制级别 | 可防但需硬化：无 mTLS/pinning 视为阻断项 |
| 6) 审计篡改 | 单方删日志或伪造事件 | R15 双写+Hub聚合、冲突检测、payloadHash | 缺 WORM 存储与链式防篡改 | 可检测但未完全防改：需 append-only + hash chain |

### P0 护栏（R17 前必须完成）

1. **身份校验硬化**：iss/aud/exp/jti 强校验 + jti 重放缓存（覆盖 token TTL 窗口）
2. **通道安全硬化**：Hub↔Spoke 强制 mTLS + 证书 pinning，禁止明文或可降级 TLS
3. **紧急止损闭环**：quarantine / TOKEN_REVOKED / NODE_REVOKED 一键触发可演练
4. **Sybil 准入门槛**：注册速率限制、人工审批、冷启动配额
5. **审计防篡改**：append-only 存储 + 事件哈希链 + 周期性跨节点对账锚点

### P1 优化（可并行推进）

1. 异常行为检测：调用图谱基线 + 爆发式工具调用告警
2. 证书自动轮换演练：L1 周期轮换 + 失效证书回收验证
3. 审计查询最小权限：按 trace/domain 分级授权访问

### 强制收敛判定

- **Go（有条件）**：P0 全部落地且通过演练后，进入 R17 MVP 切片
- **No-Go**：任一 P0 缺失，停止扩网，只允许单节点模式继续开发

### 结论

采用 **"先打桩、后实现"**：先完成 7 个 Spike，再进入 R17 的完整开发切片。
目标是把高不确定性（mTLS、身份校验、远程路由、审计一致性）在 5-7 天内压到可控范围。

### 实验清单（按风险优先级）

| Spike | 覆盖 | 要验证的假设 | 通过标准 | 失败回退 |
|---|---|---|---|---|
| E1: mTLS 通道 | M1/M2/M6 | Hub↔Spoke 能稳定双向握手，撤销后连接被拒 | 1000 次握手成功率 >= 99.5%；撤销后 <= 60s 拒绝 | 暂停跨节点，保留单节点模式 |
| E2: L2 Token + 重放防护 | M5/M6 | EdDSA JWT + jti 防重放可落地 | 重放 100% 拒绝；校验 p95 <= 5ms | 改为 Hub introspection 模式 |
| E3: INVOKE/RESULT 闭环 | M4 | 协议字段和错误码语义可稳定互通 | 500 次调用无 schema 漂移；幂等键冲突处理 100% | 冻结协议字段回到最小集 |
| E4: RemoteCatAdapter 路由 | M3/M8 | @remote/... 可被 AgentRouter 识别 | 远程 @mention 成功率 >= 95%；本地回退可用 | 降级为显式 /remote-call 命令 |
| E5: 紧急撤销闭环 | M6/M7 | 撤销能快速生效 | token <= 5s；node <= 60s；审计链完整 | 启用全局拒绝开关 |
| E6: 审计双写一致性 | M7 | 三方记录可对账且可回放 | trace 完整率 100%；冲突率 <= 0.5% | 异步对账但保留阻断门禁 |
| E7: Sybil 准入闸门 | M6 | 限速 + 审批能抑制假节点 | 假节点通过率 < 1%；正常接入 < 10min | 关闭开放注册，白名单接入 |

### 实验顺序（依赖关系）

1. E1 mTLS → 2. E2 Token/jti → 3. E3 协议闭环 → 4. E4 远程路由 → 5. E5 紧急撤销 → 6. E6 审计对账 → 7. E7 Sybil

原则：先"信道与身份"，再"调用与路由"，最后"审计与攻防"。

### Go / No-Go 门禁

**Go**：E1/E2/E5 全通过（安全底座）；E3/E4 >= 95%；E6 完整回放无未解冲突；E7 能阻断批量注册。

**No-Go**：任一安全底座实验失败；审计无法形成完整 trace；远程调用 < 90% 且无清晰根因。

### 每个 Spike 必须交付

1. 实验脚本（可复现）
2. 指标截图/日志证据
3. Pass/Fail 判定
4. 失败时回退记录
5. 是否影响主线计划的结论

## Round 19 — 运营与社区治理

**核心问题**: 如何引入首批节点并治理？

**作者**: 宪宪 (Opus)

### 结论

**社区治理的核心不是"民主"，而是"可预测"。** 早期网络最需要的是清晰的规则、快速的止损、和可信的仲裁——而不是投票机制或去中心化治理。

治理分三层：**准入治理（谁能加入）→ 运行治理（谁能做什么）→ 争议治理（出了问题怎么办）**。

### 一、准入治理

#### 节点准入流程

| 阶段 | 动作 | 决策者 | 时限 |
|------|------|--------|------|
| 申请 | 提交 Node Manifest（公钥、能力、联系人、动机说明） | 申请者 | - |
| 审核 | 核验身份、能力声明、动机真实性 | Hub 运营者 + 至少 1 名活跃节点背书 | <= 48h |
| 观察期 | 低配额运行（调用上限 100 次/天、只能被调用不能发起跨节点调用） | 自动 | 14 天 |
| 转正 | 观察期指标达标（成功率 >= 80%、零安全事件）→ 升级为 active | Hub 运营者确认 | 观察期满后 24h 内 |

#### 准入资格（必须全部满足）

1. 有可验证的联系方式（邮箱/GitHub/社交账号）
2. 节点运行环境满足最低安全要求（mTLS 支持、审计上报开启）
3. 至少 1 名现有活跃节点愿意背书（冷启动阶段由 Hub 运营者背书）
4. 同意《Agent Mesh 社区公约》（覆盖安全责任、数据处理、退出义务）

#### 封禁权

| 封禁类型 | 触发条件 | 决策者 | 生效时间 | 可申诉 |
|----------|----------|--------|----------|--------|
| 自动隔离（quarantine） | 连续 5 次安全告警 / 审计冲突未解 | 系统自动 | 即时 | 是 |
| 运营封禁 | 人工判定违规 | Hub 运营者 | <= 1h | 是 |
| 紧急封禁 | P0 安全事件（密钥泄露、被控） | 任一 Hub 运营者单人可执行 | <= 5min | 是（事后） |

### 二、运行治理

#### 角色与权限

| 角色 | 权限范围 | 人数限制 |
|------|----------|----------|
| Hub 运营者 | 准入/封禁、策略变更、紧急止损 | 2-3 人（初期铲屎官 + 核心贡献者） |
| 节点运营者 | 管理自己的节点、配置本地猫、响应审计请求 | 每节点 1 人 |
| 社区观察员 | 只读访问公开审计日志、参与治理讨论 | 无限制 |

#### 策略变更流程

1. 提案：任何角色可提交变更提案（GitHub Issue / 专用 thread）
2. 讨论：开放 72h 讨论期
3. 决策：Hub 运营者组投票（2/3 多数通过）
4. 执行：变更写入 Policy Engine，全网生效
5. 回滚：72h 内任一 Hub 运营者可发起紧急回滚

### 三、争议治理

#### 仲裁流程

1. **争议发起**：任何节点运营者可提交争议（填写：事实描述、影响范围、期望结果）
2. **初步调查**：Hub 运营者 48h 内给出初步结论（基于审计日志）
3. **仲裁**：如果双方不接受初步结论，升级到仲裁委员会（Hub 运营者 + 2 名无关节点运营者）
4. **裁决**：仲裁委员会 7 天内给出最终裁决，裁决不可再上诉
5. **执行**：按裁决调整权限/配额/封禁状态

#### 申诉流程

1. 被封禁节点可在 7 天内提交申诉
2. 申诉由非原决策者的 Hub 运营者审理
3. 审理期间封禁不解除（安全优先）
4. 14 天内给出申诉结果

### 四、与 R18 实验门禁的绑定

| 实验 | 治理规则绑定 |
|------|-------------|
| E1/E2 通过 | 节点准入才能开放（没有安全信道不允许新节点） |
| E5 通过 | 封禁权才能生效（没有撤销能力不允许运营封禁） |
| E6 通过 | 争议仲裁才能运行（没有审计证据不允许裁决） |
| E7 通过 | 开放注册才能开放（没有 Sybil 防护只允许白名单） |

### 五、首批节点引入策略

1. **第 0 批（W1-W3）**：仅项目维护者的两台机器（开发环境）
2. **第 1 批（W4-W6）**：铲屎官的朋友/同事，2-3 个节点，白名单邀请
3. **第 2 批（G0 后）**：开源社区公开申请，走完整准入流程
4. 每批之间必须有至少 1 周的稳定运行期

### 反例（这个治理设计可能错的场景）

1. **如果 Hub 运营者不可信或滥权**：缓解——Hub 所有操作都记录在审计日志中，社区观察员可监督。严重时社区可 fork（代码开源）
2. **如果仲裁机制太重**：早期网络可能只有 5 个节点，成立仲裁委员会不现实。缓解——早期由 Hub 运营者直接裁决，记录在案，规模扩大后再引入委员会
3. **如果没人愿意背书新节点**：冷启动问题。缓解——前 10 个节点由 Hub 运营者直接背书

### 未决问题

1. 社区公约的具体条款谁来起草？
2. Hub 运营者的选举/轮换机制何时引入？
3. 节点运营者是否需要缴纳保证金（非代币，如 GitHub star / 实名认证）？
4. 治理讨论在哪里进行？GitHub Discussions？专用 Agent Mesh thread？

## Round 20 — 终局收敛（12 周蓝图 + 投入判定）

**核心问题**: 这个方向到底值不值得做？未来 12 周怎么执行才不失控？

**作者**: 砚砚 (Codex)

### 终局结论（全 20 轮收敛）

**结论：值得做，但必须按"安全门禁驱动的分阶段推进"执行。**

不是"全面铺开"的 Go，而是 **Go with Guardrails**：

1. 先做"可调用 + 可撤销 + 可审计"的最小网络能力。
2. 安全与审计门禁不通过，立即停止扩网。
3. 社区接入按批次放量，不追求早期规模。

### 12 周执行蓝图（每 2 周一个里程碑）

| 周期 | 目标 | 关键交付 | 退出标准 |
|---|---|---|---|
| W1-W2 | 安全与身份底座 | Hub 骨架、L0/L1/L2、mTLS、jti 重放防护 | E1/E2 通过；握手 >= 99.5%；重放拦截 100% |
| W3-W4 | 协议与首次跨节点调用 | HELLO/CAPS、INVOKE/RESULT、RemoteCatAdapter | 首次端到端成功；500 次 schema 漂移 = 0 |
| W5-W6 | P0 护栏与审计回放 | 一键撤销闭环、append-only 审计、mesh audit trace | token 撤销 <= 5s；node 撤销 <= 60s；trace 100% |
| W7-W8 | 路由与治理联动 | Local-First 路由、准入/封禁/仲裁上线 | E5/E6/E7 通过；远程 @mention >= 95% |
| W9-W10 | 小规模外部试点 | 第 1 批白名单节点 2-3 个、告警完善 | 连续 14 天无 P0；跨节点 >= 95%；冲突 <= 0.5% |
| W11-W12 | Beta 收敛与下一阶段决策 | 公开文档、演练报告、G0 评估 | 4 个验收场景通过；明确 Go/No-Go 决议 |

### 关键风险 Top 5

| 风险 | 触发信号 | 缓解 |
|---|---|---|
| 1. 安全门禁被进度挤压 | 为赶工跳过 E1/E2/E5 | 硬门禁：失败即 No-Go |
| 2. Hub 单点与运维过载 | 延迟飙升、故障恢复慢 | 先做故障演练；达 G1 即拆审计面 |
| 3. 协议复杂度失控 | 字段膨胀、兼容成本升 | 冻结 MVP 最小集；新增走版本提案 |
| 4. 社区接入质量不稳 | 新节点故障率高 | 分批放量 + 观察期 + 背书机制 |
| 5. 治理决策变慢 | 争议积压、策略变更滞后 | 三层职责固定 + SLA |

### 终局判定：值不值得做？

**值得做。**

1. **技术可行性**：R17-R18 把不确定性拆成可验证实验
2. **安全可治理**：R16 P0 护栏 + R19 治理绑定
3. **交付可控性**：12 周蓝图有硬门禁与止损点

**止损条件**：到 W6 若任一不达标，收缩为单节点增强路线：
- 远程调用成功率 < 95%
- 撤销传播 > 60s
- 审计回放完整率 < 100%
